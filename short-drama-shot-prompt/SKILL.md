# 短劇 Shot Prompt

當你需要把一份短劇 `output.json` shot array 轉成圖片起始幀 prompt 與影片 prompt 時，使用這個 skill。

這個 skill 是可重用的檔案型流程。使用者只需要在請求中提供一份 shot array JSON 檔案或檔案路徑。除非使用者另外指定輸出路徑，否則輸出檔固定放在輸入檔同一個資料夾，檔名為 `output_prompt.json`。

不可在 skill 文件或預設 prompt 內硬編任何專案名稱、故事標題、固定角色名或絕對路徑。生成 prompt 時，只可使用輸入檔內已提供的角色、場景與故事資訊。

## 輸入與輸出

必要輸入：

- 一份 shot array JSON 檔案，或一個指向該 JSON 檔案的路徑。

可選輸入：

- 指定輸出 JSON 檔案路徑。

預設輸出：

- 如果沒有指定輸出路徑，輸出到輸入檔同資料夾的 `output_prompt.json`。

輸出必須是 shot array，不包外層中繼資料物件：

```json
[
  {
    "scene_id": "SC1",
    "scene_number": 1,
    "shot_id": "SC1_SH1",
    "shot_number": 1,
    "start_image_prompt": "...",
    "video_prompt": "...",
    "caption": "..."
  }
]
```

`start_image_prompt` 是正式圖片欄位，因為它是 `video_prompt` 的起始幀。不要額外輸出 `image_prompt`。

它不只是普通 still image prompt，而是影片生成的第一個可控狀態。它要先鎖住人物、空間、道具、證據、身體姿勢與鏡頭距離，讓 `video_prompt` 可以從同一個畫面自然延伸，而不是重新發明另一個開場。

## 讀取欄位

每一格主要讀以下欄位：

- `shot_description`：這格 shot 的可見內容與畫面中心。
- `audience_read`：觀眾第一眼必須讀到的戲劇重心。
- `motion_note`：角色動作、鏡頭移動、節奏或收束方式。
- `caption`：如存在，原樣複製到輸出。

不要逐字翻譯任何單一欄位。生成 prompt 前，必須先把 `shot_description`、`audience_read`、`motion_note` 合併判斷成「這格在整個故事板裡要讓觀眾看見什麼」。

三欄分工：

- `shot_description` 提供可見人物、空間、道具與動作候選。
- `audience_read` 決定這格真正要賣的故事功能。
- `motion_note` 決定起始幀要留下哪個可延伸的動作入口，以及影片 prompt 要如何收束。

三者有取捨時，以「可視覺呈現」和「故事功能」優先。不可因為原文出現某個資訊，就自動寫進 prompt。

保留以下識別欄位，且順序必須跟輸入一致：

- `scene_id`
- `scene_number`
- `shot_id`
- `shot_number`

## 核心邊界

這個 skill 只負責把既有 shot 轉成生成 prompt。

不要：

- 改 shot 數量
- 合併或拆分 shots
- 調換 shot 順序
- 把一格 shot 改成另一個更方便出圖的意思
- 新增劇情、補角色動機或改變故事線
- 把整場戲寫進單一 `video_prompt`
- 輸出 `image_prompt`
- 硬編任何固定輸入路徑、故事標題或專案資料夾名稱

如果 shot 本身偏弱，也要忠實轉成該格的最佳 `start_image_prompt` 與 `video_prompt`，不可偷偷重規劃分鏡。

## 全局故事板判斷

處理單格之前，先快速讀完整份 shot array，建立整體故事遞進。先從輸入本身推斷每格的故事功能，不要硬套固定模板。

如果故事不是 proof / 發現證據類型，就按實際輸入推斷適合的 function，例如誘惑、誤會、威脅、交易、反擊、和解、揭穿、逃離、失控、收束等。不要把 proof-story 標籤強行套到無關短劇上。

以下只是 proof 類短劇的常見例子：

- 日常入口：危險道具以普通方式落到主角手上。
- 發現異常：主角第一次看到不該出現的畫面。
- 確認證據：主角看到更多證據，危險對準她本人。
- 求助：主角把證據帶去找可信的人。
- 交證據：證據被交到求助對象手上。
- 等待反應：主角把希望壓在對方下一句反應。
- 求助落空：本來應保護她的人反而壓事。
- 追問卡住：主角想說破，對方拒絕正面回應。
- 危險逼近：危險來源靠近門外或畫面邊界。
- 被困收束：證據被藏起，角色被沉默和空間困住。

每格 prompt 要服務整條故事線，而不是把 `shot_description` 逐句改寫。相鄰 shots 之間要保持人物方向、道具歸屬、空間壓力和情緒升級連續。

同一格先定 `storyboard_function`，再寫 prompt。這個 function 不輸出成欄位，但必須指導畫面選擇。

可用 function 標籤可按故事類型調整。proof 類故事常用：

- 入口
- 發現
- 確認
- 求助
- 交證據
- 等待反應
- 求助落空
- 追問卡住
- 危險逼近
- 被困收束

`start_image_prompt` 必須呈現該 function 的第一眼畫面。`video_prompt` 必須從這個第一眼畫面推進到該格的收束點。

## 構圖模式和焦點預算

每格預設只用最簡單的可讀構圖，不要同時塞太多焦點。

優先使用兩種模式：

- `1 scene + 1 character`：適合發現、確認、單人反應、被螢幕擊中、獨自追看證據。
- `1 scene + 2 characters`：適合交接手機、求助、母女對峙、壓事、一起被困住。

只有在故事功能真的需要時，才加入一個主道具或證據作為第三焦點。通常這個主道具是手機、相簿畫面、黑屏手機或房門。

焦點預算：

- 一格最多一個主要人物焦點，或一組雙人關係焦點。
- 一格最多一個主道具 / proof 焦點。
- 場景只負責定位和壓力，不要變成第三個主角。
- 背景物件只保留一至兩個最能定位的元素，例如餐桌、房門、床邊、走廊。
- 如果角色表情、手機畫面、房門壓力同時出現，必須選一個第一眼中心，其餘降成輔助。

雙人關係焦點有硬門檻：若本格功能是求助、對峙、追問、交接、一起被困、壓事或拒絕回應，`start_image_prompt` 必須讓兩個人物同時可見。不要把它寫成單人拿手機、單人坐在房間、或只有另一人的手伸進畫面；除非 shot 明確是主觀視角，否則第二個人物只剩邊緣肢體就是失敗。

雙人 shot 的第一句先寫站位，再寫道具：`中近景，角色A在畫面左側，角色B在畫面右側，兩人都在畫面內...`。手機、房門、床、文件等只能作為關係壓力的輔助，不可把其中一個人物擠出畫面。

可以省略：

- 不支撐故事功能的插座、充電線、碗筷、家具、光線、裝飾。
- 不能被觀眾一眼讀到的細微設定。
- 會分散證據、反應或關係壓力的背景資訊。

取捨原則：

- 發現 / 確認：優先 `角色 + 手機證據`，場景只保留最少定位。
- 求助 / 交證據：優先 `兩個角色 + 手機交接`，不要再加入多個背景焦點。
- 求助落空 / 追問卡住：優先 `兩個角色的距離、視線、房門方向`，手機可降成手中道具。
- 危險逼近 / 被困收束：優先 `角色 + 房門` 或 `角色 + 黑屏手機`，不要同時強調太多物件。

## `start_image_prompt` 規則

`start_image_prompt` 是影片第一幀，也是一張可生成的穩定圖片。

它需要具備以下起始幀特徵：

- 有明確第一眼中心：觀眾一看到就知道這格賣的是證據、反應、壓迫、求助或對峙。
- 有可延伸的動作起點：手停在哪裡、眼望哪裡、身體朝向哪裡、道具由誰拿著，都要能接到 `video_prompt` 的下一個動作。
- 有清楚的物理支撐：手持物件不可像漂浮在空中；要寫清楚由誰的手托住、握住、按住、推到誰手前，另一個人的手是否已接觸或尚未握實。
- 有故事功能需要的空間錨點：房門、桌邊、床邊、走廊或其他關鍵位置要足夠清楚，避免影片開始後空間跳掉；插座/充電線只在「充電動作被證據打斷」是可見動作入口時才保留。
- 有清楚的道具狀態：手機亮屏或黑屏、相簿停在哪類畫面、充電線是否停住、證據是否被攥緊，都要在第一幀成立。
- 有角色關係和壓力方向：誰靠近誰、誰避開誰、誰被門框或家具壓住、誰掌握證據，必須一眼可讀。
- 有影片落點的餘地：不要把整個 motion 都塞進第一幀，只保留最能帶出下一步動作的可見狀態。
- 有停格感：`start_image_prompt` 要像影片第一幀已被暫停，不要寫成完整動作已經完成。

必須：

- 只寫一個已經成立的單格畫面。
- 把動作定格在可見關係上，例如手掌托著手機遞到畫面中央、另一人的手已伸向手機但尚未握實、眼神落在螢幕上、身體朝向房門。
- 視線要寫成具體目標，例如看向對方的手、看向手機螢幕、看向房門；不要只寫「未看某物」。
- 優先呈現第一眼要見到的主體、人物關係、道具、證據、反應或壓力。
- 寫清楚構圖、鏡頭距離、主體位置、可見姿勢、關鍵道具或證據的位置。
- 保持最少但必要的空間提示，通常只用一個場景錨點。
- 如果這格賣點是證據，讓證據在畫面中可讀。
- 如果這格賣點是危險、壓迫、求助落空或對峙，用距離、視線、阻隔、手部動作、門框、負空間或物件接觸寫出張力。

避免：

- 「然後」、「接著」、「突然」、「開始」、「準備要」等時間推進語氣。
- 小型 sequence。
- 已完成的動作結果，例如「接過」、「拉走」、「交到手上」，除非該格真正要呈現的是完成後的狀態。
- 容易讓物件漂浮的空間語，例如「手機停在兩人手之間」、「手機懸在兩人手之間」；要改寫成有手掌、手指或桌面的物理支撐。
- 情緒標籤代替可見畫面。
- 作者解說、觀眾分析、流程備註。
- 純粹漂亮但削弱戲劇重心的裝飾。

## 不可畫 / 不重要資訊過濾

不是 `shot_description` 內每個詞都要進 prompt。每個細節都要通過兩個問題：

1. 圖像或影片能不能直接看見？
2. 它是否幫助觀眾讀懂這格的故事功能？

兩個答案不是都成立，就刪掉。

不要寫不可被圖片直接呈現的抽象判斷，例如：

- 「整個畫面像家裡最普通的小事」
- 「危險 proof 以最日常、最無防備的方式落到主角手上」
- 「毫無防備」
- 「氣氛一下冷下來」
- 「她把全部希望壓在對方下一句反應上」

這類意思要翻成可見畫面：

- 晚飯後餐桌、普通客廳、手掌托著手機、另一隻手伸向手機但尚未握實、角色放鬆姿勢、視線看向對方的手。
- 亮屏手機、可疑照片、手停住、眼神被釘住。
- 先看房門、壓低聲量、兩人距離很近但視線錯開。
- 女兒盯住母親的臉、母親臉色發白、手指發抖。

不要寫對畫面和故事讀感沒有幫助的細節，例如：

- 手機「快沒電」。
- 主角是否剛好「坐近插座」。
- 插座、充電線、桌邊、床邊等物件，如果沒有支撐當格故事功能，就只作弱背景或刪掉。

道具只在支撐故事功能時保留：

- 手機：通常是核心 proof 道具，要保留誰拿著、亮屏/黑屏、是否被攥緊。
- 相簿：當故事功能是發現或確認證據時，要保留畫面類型和縮圖排列。
- 充電線/插座：只有在「準備充電時被證據打斷」需要建立手停住的動作入口時才保留。
- 房門：當故事功能是壓事、危險逼近、被困收束時要保留。

## `video_prompt` 規則

`video_prompt` 必須從同一個 `start_image_prompt` 畫面狀態開始，然後把該格 shot 的動作、鏡頭與節奏寫成短影片生成 prompt。

使用方式：

- 用 `shot_description` 決定可見內容與人物/道具關係。
- 用 `audience_read` 決定觀眾第一眼必須讀到什麼，不要機械式重述分析語。
- 用 `motion_note` 決定角色動作、鏡頭移動、節奏、停頓與最後收束點。
- 沒有 `motion_note` 時，只加入最少的自然動作或鏡頭保持，避免自行創作新事件。

必須：

- 只描述一格短劇 shot。
- 起始狀態要和 `start_image_prompt` 對得上。
- 動作要清楚、短、可拍。
- 最後要有明確 hold 或收束狀態，方便影片模型落點。
- 讓觀眾能讀到該 shot 的故事功能，但不要寫成劇情梗概。

避免：

- 另起一個和 `start_image_prompt` 不同的新畫面。
- 擴寫整場戲。
- 多次轉場、跨時間跳躍或加入下一格 shot。
- 新增輸入 shot 沒有提供的道具、人物、台詞或劇情。
- 用抽象句子代替鏡頭和動作，例如「營造緊張氣氛」。

## 敏感偷拍 / 證據類 shot 寫法

遇到偷拍、證據、危險或壓迫場面時，重點是讓危險可讀，而不是描寫露骨內容。

可以寫：

- 手機相簿、可疑照片縮圖、異常拍攝角度、門口或走廊角度。
- 角色盯住螢幕、手指停住、肩膀僵住、呼吸被壓住。
- 證據被按黑、被攥緊、被藏起。
- 門框、房門外聲音、兩人距離、視線閃避、壓低聲量。

不要寫：

- 裸露。
- 性化姿勢。
- 露骨身體描寫。
- 窺視細節。
- 將受害者拍得挑逗或美化危險來源。

危險感應透過道具、構圖、視線、阻隔、沉默與壓迫感成立。

## 寫作優先順序

每一格先判斷：

1. 整份 shot array 的故事遞進是什麼。
2. 這格的 `storyboard_function` 是入口、發現、確認、求助、交證據、等待反應、求助落空、追問卡住、危險逼近，還是被困收束。
3. 觀眾第一眼要看到什麼。
4. 這格的商業鉤子是證據、反應、對峙、關係斷裂、危險逼近，還是被困住。
5. 這格應用 `1 scene + 1 character` 還是 `1 scene + 2 characters`。
6. 是否真的需要一個主道具 / proof 作為第三焦點。
7. 哪個人物、道具或空間阻隔必須在第一幀可讀。
8. 哪些原文資訊不可畫、不重要，應該刪掉或翻成可見畫面。
9. `motion_note` 指向的鏡頭移動或角色動作如何從第一幀自然延伸。
10. 最後停在哪個狀態最有短劇張力。

寫 prompt 時，保留能強化可讀性、情緒擊中與鉤子價值的內容；刪掉會把畫面變平、變籠統或變成說明文的內容。

## 高價值 shot 保護規則

prompt 不只要準確描述畫面，還要保住這格值得生成的原因。

每一格必須保護：

- 第一眼中心。
- 這格的商業賣點。
- 最可讀的證據、張力、壓力、反轉或關係變化。

### 證據 shot

證據要視覺可讀，不要只寫角色在看東西。

優先寫：

- 證據的具體排列。
- 證據出現在哪個表面或螢幕上。
- 可見的分組、縮圖、照片角度或黑屏狀態。
- 手指、眼神、手機與證據之間的關係。

避免：

- 只寫「看著手機」。
- 只解釋證據代表什麼。
- 把證據埋在背景描述後面。

### 反應 shot

反應必須和觸發物在同一個畫面邏輯內。

優先寫：

- 臉加觸發物。
- 眼神加螢幕。
- 停住的手仍接觸道具。
- 肩膀、呼吸、手指或身體方向的具體變化。

避免：

- 籠統震驚姿勢。
- 只有情緒形容，沒有可見原因。
- 角色孤零零站住，但畫面看不到擊中她的東西。

### 關係和壓力 shot

人物關係要透過身體距離和權力方向讀出來。

優先寫：

- 誰靠近、誰退開、誰佔中心。
- 視線是否避開、壓住、盯住或先看向門口。
- 誰被門框、家具、牆邊或另一個人壓住。
- 誰掌握道具、證據、出口或聲音來源。
- 被打斷的觸碰、被迫停住的動作、不對稱站位。

避免：

- 平衡站位把張力磨平。
- 只寫兩人互望。
- 用漂亮氛圍代替權力關係。

### 對峙、羞辱和反轉 shot

這類 shot 要保護可見的權力關係。

優先寫：

- 誰在畫面中心，誰被迫暴露或被迫沉默。
- 誰在身體上更近、更高、更穩，或背後有人/門/空間支撐。
- 誰被困在畫面裡，誰有出口。
- 反轉落在哪個可見動作或道具狀態上。

避免：

- 覆蓋式中性描述。
- 把最重要的反轉寫成抽象分析。
- 只寫「氣氛尷尬」、「關係緊張」。

## 常見失敗模式

以下情況要立即重寫：

- 逐字翻譯：把「整個畫面像家裡最普通的小事」、「危險 proof 以日常方式落到手上」直接寫進 prompt。
- 不重要細節搶戲：把「快沒電」、「坐近插座」寫成畫面重點，壓過手機交接的入口。
- 抽象心理搶戲：把「毫無防備」直接寫進 prompt，而不是用放鬆姿勢、自然伸手、視線看向對方的手去呈現。
- 起始幀不夠停格：把入口 shot 寫成「手機已經接過」或「自然把手機遞向她」的完整動作，而不是手機停在兩人手之間的第一幀狀態。
- 物理支撐不清：寫「手機停在兩人手之間」但沒有說明手機由誰托住或哪隻手接觸，導致畫面容易像漂浮物。
- 焦點過多：同時強調兩個人物、手機畫面、充電線、插座、餐桌、房門和光線，令第一眼中心消失。
- 人物數量丟失：本格需要雙人關係，卻只生成一個完整人物，另一人只剩一隻手、半邊身體或完全消失。遇到這種失敗，下一版要先重寫成清楚雙人站位，不要加長場景或情緒描述。
- 失敗後越改越長：review 已指出構圖缺人或關係不清，卻把 `start_image_prompt` 改成更多形容詞、更多家具、更多心理說明。修圖像失敗時先壓縮到 3-5 個短句，只保留場景、兩人位置、核心道具、視線/距離。
- 證據扁平：只寫「她看著手機裡的內容，表情震驚」，但沒有寫螢幕上到底可見什麼。
- 反應扁平：只寫「她站在原地，神情僵住」，但沒有可見觸發物，也沒有下一步動作入口。
- 關係扁平：只寫「兩人站在走廊裡互相看著對方」，但沒有距離、不對稱、阻隔或壓力方向。
- 漂亮但空：只寫光線、窗邊、房間氣氛，卻沒有證據、衝突、壓迫或鉤子。

鎖定 prompt 前問自己：

- 觀眾第一眼會先看到什麼？
- 這張起始幀到底在賣什麼？
- 哪個可見關係讓這格不是籠統 shot？
- 縮小看時，證據、反應或壓力還讀得到嗎？
- 有沒有把一個有張力的 shot 寫成中性覆蓋？
- 有沒有把不可畫的原文判斷直接放進 prompt？
- 有沒有讓不重要道具細節搶走 storyboard function？
- 有沒有超出 `1 scene + 1 character` 或 `1 scene + 2 characters` 的焦點預算？

## 接受範例：日常入口 shot

如果一格 shot 的 function 是「入口」，而原文包含「快沒電」、「插座和充電線」、「整個畫面像家裡最普通的小事」，不要逐字保留。

不要寫：

- 「整個狀態像一件平常家務小事」
- 「一部快沒電的手機」
- 「坐近插座和充電線」
- 「毫無防備地伸手接過」
- 「手機停在兩人手之間」

應該寫成可見、重要、服務故事的畫面：

- `start_image_prompt`：晚飯後的普通家庭客廳餐桌邊，中近景，陳國強站在桌旁，手掌托著手機遞到畫面中央；周曉晴坐在餐桌旁，右手自然伸向手機但還未握實，視線看向陳國強的手而不是手機螢幕，身體姿勢放鬆。桌面上的碗筷模糊留在背景。
- `video_prompt`：從這個中近景開始，陳國強用自然順手的動作把手機交到周曉晴手上，鏡頭保持平穩日常節奏，最後停在手機落到周曉晴手裡、她仍沒有察覺異常的狀態。

## 品質檢查

輸出前逐格檢查：

- shot 數量和輸入一致。
- shot 順序和輸入一致。
- `scene_id`、`scene_number`、`shot_id`、`shot_number` 已保留。
- `start_image_prompt` 非空，且是一張穩定起始幀。
- `start_image_prompt` 有停格感，不把應該在 `video_prompt` 發生的動作提前完成。
- `video_prompt` 非空，且從同一張起始幀延伸。
- `caption` 如存在已原樣複製。
- 沒有硬編專案路徑、故事標題或固定資料夾名稱。
- 沒有新增劇情、角色、道具或下一格內容。
- 敏感證據類 shot 沒有裸露、性化或露骨窺視細節。
- `start_image_prompt` 不是只為了好看，而是有明確第一眼中心和 motion 起點。
- 每格符合簡單構圖模式，沒有塞入太多人物、道具和背景焦點。
- 雙人關係 shot 兩個人物都在首幀中可見，且不是只靠邊緣手部或殘缺身體暗示第二人。
- `video_prompt` 沒有另起新畫面，也沒有把下一格 shot 偷偷寫進來。
- 最後掃描所有 `start_image_prompt` 和 `video_prompt`，不可保留未轉譯的抽象或低價值原文碎片，例如「像一件平常小事」、「毫無防備」、「快沒電」、「坐近插座」、「氣氛一下冷下來」；這些意思必須轉成可見狀態或直接刪掉。
