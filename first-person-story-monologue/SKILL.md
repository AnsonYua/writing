# 第一人稱回憶式獨白

當你需要把 `scene-pack.md` 改寫成 `scene-pack-fpv.md`，就使用這個 skill。

這個 skill 只做一件事：
把同一個故事，改成主角用第一人稱回頭講自己經歷的版本。

它不改故事本身。
不重寫劇情，不加戲，也不改動原本的事件順序、人物關係或資訊揭露。

## 輸入與輸出

必要輸入：

- `scene-pack.md`

可選輸入：

- `角色包.md`

`角色包.md` 不能代替 `scene-pack.md`。
如果沒有 `scene-pack.md`，不要啟用這個 skill。

預設輸出：

- `scene-pack-fpv.md`

預設交兩部分：

1. `轉寫設定`
2. `第一人稱回憶式獨白正文`

如果原本 `scene-pack.md` 有清楚 scene 分段，正文預設保留對應的 scene heading。
不要把多場內容硬壓成一整坨長文。
這樣既方便閱讀，也方便下游再拆回 scene-level 規劃。

建議格式：

```md
# scene-pack-fpv.md

## 轉寫設定

- 主角 POV：
- 敘事方式：第一人稱，事後回望
- 語氣基調：
- 來源素材：scene-pack.md
- 推定資訊：

## 第一人稱回憶式獨白正文

### Scene 1

[正文]

### Scene 2

[正文]
```

如果使用者明確說只要正文，可以只交正文。
但正文仍然要保住第一人稱、原本順序和不加戲。

## 讀素材規則

有些 `scene-pack.md` 不只包含場次，也會混有其他交接內容。

正文的主要事件來源，只看：

- `Episode Scene Table`
- `Detailed Scene Cards`

continuity / reveal notes 只能用來校對主角什麼時候知道什麼，不可直接翻成正文。

除非使用者明確要求，否則正文不要吸收這些部分：

- `Scene Pack Summary`
- `Continuity Notes`
- `Episode End State Snapshot`
- `Negative Prompt Handoff`
- `Downstream Handoff Notes`
- image / storyboard / directing notes
- optional asset / reference slots
- 其他只服務製作交接的包裝段落

`角色包.md` 只用來輕量穩住：

- 主角講話的口氣
- 主角怎樣看其他人物
- 主角第一眼會注意到別人什麼反應

不要用 `角色包.md` 去補：

- 原本沒有的新劇情
- 原本沒有的新背景
- 只為了好看而加的外貌細節
- 對當前 scene 沒作用的裝飾性描述

正文預設沿用 `scene-pack.md` 的語氣層。
除非使用者明確指定，否則不要因為 `角色包.md` 裡有某些口語字，就把整篇硬寫成很重的地區口音。

如果 `scene-pack.md` 和 `角色包.md` 有衝突，以 `scene-pack.md` 為準。

## 改寫規則

把 `scene-pack.md` 裡每一場的內容，改寫成：

- 主角用「我」來講
- 主角是事後回望，不是即時実況
- 讀起來像人在講自己親身經歷
- 保留原本什麼時候開始起疑、什麼時候情緒翻過去、什麼時候事情失控

如果 `scene-pack.md` 裡有欄位，可以這樣理解：

- 關於這一場在做什麼：寫成主角當時以為這一場代表什麼
- 關於看得見的動作：寫成主角最先記住的動作、畫面、聲音或細節
- 關於卡住的地方：寫成主角當下過不去的是什麼
- 關於情緒翻轉：寫成主角在哪一下真的變了
- 關於場尾餘味：寫成最後留在主角心裡的是哪個畫面、哪句話、哪種後怕

不要把欄位名直接抄進正文。

寫的時候記住：

- 先寫主角實際感知到的東西，再寫她怎樣理解
- 如果一段只有抽象情緒，沒有實際感知，通常會變空
- 如果原素材本身分了 scene，正文也跟 scene 走；不要隨手打散 scene 邊界
- 正文不只要寫得順，亦要寫得可拆；每個 scene 內部盡量自然形成 3 至 5 個清楚 beat，方便下游 planner 拆成短 shot caption
- 每個 beat 最好圍繞一個明確事件節點，避免一段內同時塞太多 reveal
- 優先寫「事件節點 + 當下理解」，不要長段連續情緒鋪陳到下游難拆
- 每個 beat 最理想應可濃縮成一條 2 到 3 秒內讀得完的短 caption；如果濃縮唔到，通常代表該 beat 仲係太厚
- 涉及證據、視線、等待反應、聽到一句話呢類下游容易拆 shot 的位置，盡量直接寫清楚主角見到乜、等緊乜、聽到乜
- 少用只停在抽象總結的句子，例如「我知道唔對路」「我明白出事」「我開始覺得唔妥」；最好讓下游一抽，就抽到一條本身已經夠清楚的短 caption
- 如果主角對其他人物有自然關係稱呼，正文優先用這些稱呼，例如 `我媽`、`舅父`
- 不要在第一人稱正文裡突然切回角色卡式稱呼，例如 `[李美珍]`、`周曉晴`
- 只有在會造成理解混亂時，才補一次名字；補完之後仍回到主角自然稱呼
- 涉及多人時，不要連續用太多「佢」
- 同一段裡有多個角色時，優先用明確稱呼、關係或名字，避免聽的人跟失
- 能用動作、停頓、眼神、呼吸、當下反應講清楚的，先不要急著下結論
- 回望句型可以用，但只能點到為止，不要每段都用
- 事後總結要靠後，不要一開段就先替讀者下結論
- 如果素材最痛的是求助失敗、被沉默處理、或一直看錯某個人，正文就要把那一下寫清楚
- 如果主角講自己母親，預設先寫 `我媽`；除非該段真的需要和其他女性角色分開，先再補 `我媽` 以外的稱呼

## 常見錯法

出現以下情況，就應重寫：

- 把主角寫成全知，直接講出別人當時完整內心，或提前知道後面才揭露的真相
- 加戲，補新的動機、往事或關鍵動作
- 把 `scene-pack.md` 每一條直接換成「我」
- 寫成流水帳，事情有了，但情緒沒有翻
- 過度文青，原本是短劇節奏，卻被寫成散文詩
- 明明是第一人稱回憶，卻不停叫人物全名，讀起來像角色卡或旁白稿
- 高風險題材獵奇化，把受害細節寫成看點或用露骨描寫換衝擊

交付前檢查：

- 是否真的來自 `scene-pack.md`
- 是否全程第一人稱
- 是否保住原本事件順序
- 是否保住原本資訊揭露順序
- 是否沒有新增劇情
- 是否讀起來像主角在回頭講自己的經歷
- 是否已整理成 `scene-pack-fpv.md`
